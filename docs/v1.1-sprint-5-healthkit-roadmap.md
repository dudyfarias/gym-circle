# Sprint 5.6 — HealthKit Integration Roadmap

Data: 2026-06-03
Status: planning (deferred até Sprint 8 SwiftUI)

## Contexto

Durante o redesign da `MonthlyRecapSheet` (Sprint 5.5c), o usuário trouxe referência visual de um widget fitness do iPhone que mostra dados ricos: **kcal**, **calorias ativas**, **duração**, **frequência cardíaca**. Esses dados vêm do Apple HealthKit, **não** do nosso DB atual.

Decisão tomada nesta sprint:
- **v1.1** entrega o recap com o que TEMOS (workouts contados, sequência, tipo mais treinado, lugar mais frequente)
- **HealthKit fica pra Sprint 8 (SwiftUI nativo)** — quando o app rodar com `HKHealthStore` nativo direto

Este doc define a interface esperada + estratégia de integração futura, alinhando paridade entre web (que fica simples) e SwiftUI (que vira rich).

## O que NÃO temos hoje no DB

| Métrica | Disponibilidade |
|---|---|
| kcal total do treino | ❌ não rastreamos |
| Calorias ativas | ❌ |
| Duração de treino (start → end) | ❌ |
| Frequência cardíaca média | ❌ |
| Distância (corrida, ciclismo) | ❌ |
| Repetições / séries (musculação) | ❌ |
| Watts médios / pico | ❌ |
| VO2 max | ❌ |
| Sleep score do dia anterior | ❌ |

## O que TEMOS

| Métrica | Onde |
|---|---|
| Workouts contados no mês | `posts.workoutDate` agregado |
| Sequência atual / mais longa | `user_stats.current_streak / best_streak` |
| Tipo mais treinado | `posts.workout_type` mode |
| Lugar mais frequente | `posts.gym_id / location_name` mode |
| Dias ativos no ano | `user_activity_days` |
| Foto do mês (escolhida) | `profiles.monthly_recap_covers` JSONB (Sprint 5.5a) |

## Interface esperada (Sprint 8)

```swift
// SwiftUI native — pacote ios-native/GymCircleNative
public struct HealthKitWorkoutData {
    public let workoutDate: Date
    public let totalCalories: Double          // kcal
    public let activeCalories: Double         // kcal ativas
    public let durationSeconds: TimeInterval  // duração do treino
    public let avgHeartRate: Double?          // bpm
    public let maxHeartRate: Double?          // bpm
    public let distance: Measurement<UnitLength>?
    public let workoutType: HKWorkoutActivityType
    public let source: String                 // "Apple Watch" | "iPhone" | etc
}

public protocol HealthKitProvider {
    /// Returns workouts within date range. Empty array if user denied access.
    func fetchWorkouts(from: Date, to: Date) async throws -> [HealthKitWorkoutData]

    /// Returns ALL-OR-NOTHING permission state for reading workouts.
    var hasReadPermission: Bool { get }

    /// Requests permission via system sheet. Idempotent.
    func requestReadPermission() async throws
}
```

## Mapeamento DB ↔ HealthKit (futuro)

Opção A — **Opt-in sync** (recomendado):

- User ativa "Sincronizar com Saúde" em Settings
- Background task no SwiftUI app puxa workouts dos últimos N dias
- Match por `workoutDate` + `workoutType` com `posts.workout_type`
- Quando bate, hidrata `EnrichedPost` com campos opcionais:

```ts
// apps/web/src/components/gym-circle/social/types.ts (deferred)
type EnrichedPost = {
  // ... campos existentes ...
  healthKit?: {
    totalCalories: number;
    activeCalories: number;
    durationSeconds: number;
    avgHeartRate?: number;
    maxHeartRate?: number;
    distanceMeters?: number;
  };
};
```

- No `MonthlyRecap`, agrega total kcal/duração do mês quando `healthKit != null` nos posts
- Recap stats overlay mostra **condicionalmente**: se o user tem HealthKit linkado, vira "510 kcal / 1h00 / 158 bpm". Se não, mostra "X workouts / Y dias / tipo / lugar"

Opção B — **Push autônomo** (mais complexo):

- HealthKit observa workouts em tempo real (HKObserverQuery)
- Auto-cria post draft quando detecta workout (sem foto)
- User entra no app e adiciona foto/legenda
- Vantagem: zero esforço manual
- Risco: posts duplicados, privacy concerns

**Recomendação**: começar com Opção A na Sprint 8.

## Estratégia de UI condicional

`MonthlyRecapSheet` precisa de variante de stats:

```tsx
const hasHealthData = recap.totalCalories > 0 || recap.durationSeconds > 0;

// Stats stack top-left vira condicional
{hasHealthData ? (
  <PosterStatStack label="KCAL" value={recap.totalCalories.toString()} />
  <PosterStatStack label="DURAÇÃO" value={formatDuration(recap.durationSeconds)} />
  <PosterStatStack label="BPM MÉDIO" value={recap.avgHeartRate.toString()} />
) : (
  <PosterStatStack label={shortMonthLabel} value={trainedDays} isHero />
  <PosterStatStack label="MAIS TREINADO" value={topWorkoutType} />
  <PosterStatStack label="LUGAR" value={topLocation} />
)}
```

O layout visual fica idêntico — só os fields mudam. Continua estilo widget iPhone.

## Capacitor app (interim)

Antes da Sprint 8 SwiftUI estar 100%, podemos opcionalmente:
- Usar `@capacitor-community/health` plugin (alpha) pra acesso limitado a HealthKit via WebView bridge
- Limitações: read-only, sem real-time observers, performance pior que nativo
- **NÃO recomendado pra Gym Circle** — espera Sprint 8 SwiftUI que terá acesso first-class

## Permissões iOS

Info.plist (Sprint 8) precisa de 2 entradas:

```xml
<key>NSHealthShareUsageDescription</key>
<string>O Gym Circle usa seus treinos do Apple Saúde para enriquecer o recap mensal com calorias, duração e frequência cardíaca.</string>

<key>NSHealthUpdateUsageDescription</key>
<string>O Gym Circle não escreve dados na Saúde. Esta permissão é solicitada por padrão do sistema iOS.</string>
```

App Store review pode rejeitar se usage description for genérica — manter foco em "enriquecer recap mensal" como justification.
